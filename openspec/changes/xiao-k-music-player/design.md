## Context

小 k 已由 `layouts/partials/chat-widget.html` 注入页面，前端逻辑集中在 `static/chat-ui.js`，聊天请求由 VPS 本地 `server/chat-api.mjs` 提供。音乐播放器需要嵌入现有聊天面板，但音乐上游和播放失败不应影响聊天主链路。

原始草案位于 `docs/superpowers/specs/xiao-k-music-player.md`，本 change 保留该文件作为历史参考，不修改、不删除。后续实现以本 OpenSpec change 为准。

## Goals / Non-Goals

**Goals:**

- 在聊天面板内提供可展开的音乐播放器，不跳转页面。
- 通过独立 Node 服务代理音乐搜索和播放地址，避免前端直接依赖上游细节。
- 支持用户手动搜索播放，也支持小 k 用播放标记触发自动加入队列。
- 对上游不可用、歌曲不可播、浏览器自动播放拦截做温和降级。
- 保持聊天、留言墙、博客静态页面在音乐服务异常时继续可用。

**Non-Goals:**

- 不做用户登录、收藏、歌词、历史播放持久化或本地曲库。
- 不代理整段音频流，只返回上游播放 URL 并由浏览器 `<audio>` 播放。
- 不做 STT/TTS 语音对话。
- 不改变 `/api/chat` 的聊天请求契约。

## Decisions

### D1: 音乐能力使用独立代理服务

音乐代理由后续 `server/music-api.mjs` 提供，默认监听 `127.0.0.1:8789`。它只负责转发上游、标准化响应、短期内存缓存和错误降级，不存储用户数据。

备选方案是把音乐接口合并进 `server/chat-api.mjs`，但会把聊天主链路与音乐上游耦合在一起。独立服务更便于部署、重启和故障隔离。

### D2: 前端播放器状态只保存在页面内存

播放队列、当前歌曲、播放进度和音量控制由 `static/chat-ui.js` 管理。刷新页面后不恢复队列，也不自动恢复播放。

这样可以避免引入新的持久化规则和隐私边界；首版只解决“当前会话里听歌”的体验。

### D3: 小 k 联动通过文本标记完成

assistant 回复可在末尾包含 `{{play:歌名|歌手}}`。前端只在回复完整结束后解析一次，使用 `歌名 歌手` 搜索，优先选择歌名精确且歌手匹配的结果，失败再使用第一条结果。

备选方案是新增结构化工具调用，但当前聊天前端和服务端已经以纯文本流式回复为主。文本标记实现成本更低，也能保持后续回滚简单。

### D4: 原始播放标记不直接展示给用户

前端渲染 assistant 消息时隐藏原始 `{{play:...}}`，替换成“已加入播放器”之类的轻量提示。历史消息重新渲染时只显示提示，不再次触发搜索或播放。

这能避免用户看到内部协议，同时防止刷新页面后重复播放。

### D5: CORS 与缓存沿用轻量服务策略

音乐代理使用 `MUSIC_ALLOWED_ORIGINS` 控制 CORS，默认复用聊天/留言墙允许的站点白名单。搜索结果缓存 60 秒，播放 URL 缓存 10 分钟；上游异常和错误结果不缓存。

缓存只存在于进程内，不作为业务状态，不需要迁移或清理流程。

## Risks / Trade-offs

- 网易云或自管上游不可用 -> 代理返回 `music_unavailable`，前端只在音乐区域提示，不阻断聊天。
- 歌曲无版权或播放 URL 为空 -> 当前歌曲标记为不可播，队列保留并允许用户尝试其他结果。
- 搜索结果错配 -> 使用“歌名精确 + 歌手包含”的优先匹配降低概率，但首版不承诺完全准确。
- 浏览器自动播放限制 -> 捕获 `audio.play()` 失败，展开播放器并提示用户点击播放。
- 音频 URL 跨域或 mixed content -> 首版不代理音频流，播放失败时降级提示。

## Migration Plan

1. 先完成本 OpenSpec change，并通过 `openspec validate xiao-k-music-player --strict`。
2. 后续实现阶段按 `tasks.md` 顺序完成后端代理、部署配置、前端播放器、聊天联动和 prompt/建议词更新。
3. 部署时新增音乐服务 systemd 单元与 Nginx `/api/music/` 反代，不替换现有 `/api/chat`。
4. 如音乐服务上线后异常，可停用 Nginx `/api/music/` 反代或停止音乐 service；聊天主链路不需要回滚。

## Open Questions

- `MUSIC_API_UPSTREAM` 的线上默认值在实现阶段确认，OpenSpec 只固定通过环境变量注入。
- 播放失败提示文案和提示卡片具体样式在实现阶段对齐现有聊天视觉。
